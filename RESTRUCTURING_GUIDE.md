# Study Buddy Restructuring Guide: Scaling to an API

This guide outlines the conceptual changes needed to separate the **Ingestion (Producer)** and **API/Query (Consumer)** layers of the Study Buddy (GraphRAG) project.

## 1. The Producer: `main.py` (Ingestion & Indexing)
Transform `main.py` into a standalone ingestion script that processes data and prepares the knowledge graph.

### Key Changes:
*   **Persist the Graph:** The `Neo4jPropertyGraphStore` already persists the graph structure to Neo4j.
*   **Persist Community Summaries:**
    *   Currently, `GraphRAGStore.community_summary` is a Python dictionary in memory.
    *   **Action:** At the end of `main.py`, serialize `index.property_graph_store.community_summary` to a file (e.g., `data/community_summaries.json`).
*   **Remove Query Logic:** Delete the `GraphRAGQueryEngine` setup and the `query_engine.query(...)` calls. `main.py` should exit after the graph and summaries are ready.

## 2. The Consumer: `app.py` (FastAPI & Querying)
Transform `app.py` into a persistent web server that connects to the existing Neo4j data.

### Key Changes:
*   **Initialize FastAPI:** Use the `fastapi` library to create a REST API.
*   **Reconnect to Neo4j (No Re-indexing):**
    *   Instantiate `Neo4jPropertyGraphStore` with the same credentials.
    *   Use `PropertyGraphIndex.from_existing(property_graph_store=...)` instead of passing `nodes`. This avoids regenerating triplets and embeddings.
*   **Load Community Summaries:**
    *   Read the `data/community_summaries.json` file generated by `main.py`.
    *   Inject this data back into your `GraphRAGStore` instance.
*   **Persistent Query Engine:**
    *   Initialize `GraphRAGQueryEngine` once during server startup.
    *   Create a `POST /query` endpoint that takes a user's question and returns the answer from the query engine.

## 3. Workflow Comparison

| Feature | Current (Script Mode) | Target (API Mode) |
| :--- | :--- | :--- |
| **Execution** | Runs manually for every query. | **`main.py`**: Runs once to ingest.<br>**`app.py`**: Runs 24/7 as a service. |
| **Data Lifecycle** | Summaries are lost on exit. | Summaries are saved to disk/Neo4j. |
| **Embedding Cost** | Regenerated every run (High cost/time). | Generated once in `main.py` (Zero cost in `app.py`). |
| **Interface** | Terminal output. | JSON API (ready for a React/Vue frontend). |

## 4. Required Class Updates (`src/core_classes.py`)
To make this work, you'll need to add a method to `GraphRAGStore` that allows external injection of community summaries:

```python
# Conceptual change only
def load_community_summaries(self, summaries: dict):
    self.community_summary = summaries
```
